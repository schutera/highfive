# Nest-hole grid reference (issue #165)
#
# SUPERSEDED. The original hand-measured pixel boxes below were taken on a
# single 791x528 reference frame and described a 4-bee-type x 3-nest layout.
# The physical laser-cut block actually has 4 nests per bee type (a 4x4 grid),
# and ESP captures arrive at VGA (640x480) or UXGA (1600x1200) — never 791x528 —
# so absolute pixel boxes don't transfer.
#
# The live detector (`image-service/services/hole_detection.py`) now finds holes
# dynamically with OpenCV HoughCircles and labels bee type *by measured
# diameter*, so it needs no per-resolution calibration. Its only fixed asset is
# a NORMALIZED fallback grid (fractions of width/height, resolution-independent),
# used when HoughCircles fails. That fallback lives in code as
# `_FALLBACK_X_FRACS` / `_FALLBACK_Y_FRACS` / `_FALLBACK_RADIUS_FRAC` and SHOULD
# be recalibrated against a current capture from a representative module.
#
# Normalized 4x4 fallback (as fractions; mirror of the code constants):
#   columns (nests, x):     0.30  0.44  0.58  0.72
#   rows    (bee types, y): 0.22  0.42  0.62  0.82
#   radius (of width):      0.06
#
# Bee types are ordered by ascending hole diameter (homepage BEE_TYPES):
#   2 mm black-masked < 3 mm resin < 6 mm leafcutter < 9 mm orchard.
#
